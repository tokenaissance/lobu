I keep seeing the same question framed the wrong way:

**Should agent memory live in the filesystem or in a database?**

I do not think that is the right choice.

My background is in data systems. In a normal company, you do not ask whether the business should run on analysts' notebooks or on the warehouse. You need both.

- operational systems run the business
- a central warehouse accumulates durable facts from those systems
- analysts use SQL and Python to make sense of the data
- the cleaned result flows back into tables that the rest of the company can use in BI tools

The same pattern shows up in agent systems.

- the **filesystem** is where the agent does temporary work
- the **memory layer** is where the organization stores durable knowledge

That is the model behind Lobu's [Memory docs](/getting-started/memory/).

## The data warehouse analogy

A company usually has many operational systems: CRM, support, billing, product events, internal tools.

Those systems are great at running transactions. They are not great at answering broad organizational questions.

So teams build a warehouse.

A useful mental model is this:

- **OLTP systems** are the shop floor
- **the warehouse** is the stockroom
- **analysts** are the people turning raw inventory into something the business can actually use

The warehouse becomes the central place where the organization can ask:

- what changed?
- what is the current truth?
- what should everyone else be able to see?

## Agents have the same split

An agent also has two kinds of state.

First, there is the messy working state:

- downloaded PDFs
- screenshots
- temporary scripts
- CSV exports
- notebooks
- intermediate results
- generated reports that may or may not matter later

That belongs in the filesystem.

Second, there is durable knowledge:

- who owns a project
- what a customer prefers
- which risk is blocking a rollout
- what changed since last week
- which facts should another agent or user be able to recall later

That belongs in a shared memory layer.

If you try to put everything into the filesystem, you get isolated sandboxes full of local context that nobody else can use.

If you try to put everything into the database, you get a junk drawer full of temporary artifacts that should never have been promoted to memory in the first place.

## What analysts do, agents now do too

This is the part that feels familiar to me.

An analyst might:

1. pull raw data from business systems
2. use SQL or Python to clean and analyze it
3. write the result back into Snowflake tables
4. let business users explore that curated result in a BI tool

An agent often does the same shape of work:

1. pull files and data from tools, APIs, or conversations
2. use code and tools to inspect, transform, and analyze them
3. write the durable result into organizational memory
4. let other agents, operators, or workflows use that updated knowledge later

The metaphor I like is simple:

> The filesystem is the analyst's notebook. The memory layer is the warehouse.

You would never ask a company to run its reporting directly from a pile of temporary notebooks. And you would not ask analysts to do all of their work by editing warehouse tables by hand.

Agents need the same separation.

## Why filesystem memory is not enough

Filesystem memory is excellent for ephemeral work. It is close to the agent's tools, cheap to use, and naturally fits active problem solving.

But it breaks down when you need:

- memory shared across agents
- recall across sessions or threads
- structured facts instead of loose files
- auditability and correction
- freshness from connectors or scheduled updates

A workspace can tell you **what the agent touched**.

A memory layer should tell you **what the organization now knows**.

That is a different job.

## Why the memory layer should not replace the filesystem

The opposite mistake is also common.

Not every artifact deserves promotion into shared memory.

A half-finished CSV, a temporary chart, a debug script, or a downloaded PDF is often just working state. Useful now, irrelevant later.

Good agents should be able to think messily and remember cleanly.

That means:

- keep temporary work in the filesystem
- write stable meaning into the memory layer

## How this maps to Lobu

Lobu makes this split explicit.

- Every agent has a local workspace.
- If `MEMORY_URL` is not set, Lobu uses `@openclaw/native-memory` and memory stays local to that workspace.
- If `MEMORY_URL` is set, Lobu uses `@lobu/owletto-openclaw` and writes durable memory into Owletto.

Owletto plays the role of the organizational knowledge layer:

- shared across agents and sessions
- structured as entities, relationships, and events
- searchable by more than file paths and grep
- inspectable and correctable by operators

## What should go where?

A simple rule of thumb:

### Keep in the filesystem

- raw source files
- temporary code
- intermediate analysis outputs
- drafts
- screenshots
- one-off transformed data

### Write to the memory layer

- customer facts
- user preferences
- project state
- blockers and ownership
- recurring operational knowledge
- decisions worth recalling later
- references to the evidence behind those facts

## The short version

If your agent is acting like an analyst, give it a notebook.

If your organization wants lasting knowledge, give it a warehouse.

That is why serious agent systems usually need both.
