# Security

This document explains Lobu's security model at a practical level: what is isolated, what is trusted, and how deployment mode changes the controls.

## Threat Model

Assume:
- The agent can execute untrusted code (generated by an LLM) and fetch untrusted dependencies.
- Skills and MCP servers can be third-party code/services.
- Users in different Slack channels/DMs must not be able to read/write each other's workspaces or secrets.

Goal:
- Contain compromise to a single sandboxed worker/session.
- Centralize secrets and outbound access decisions in the gateway.

## Kubernetes

In Kubernetes mode, the gateway orchestrates per-session worker pods and persistent volumes.

Controls typically used:
- **Pod isolation**: workers run in their own pods with constrained resources.
- **NetworkPolicies**: workers are not publicly reachable and egress is constrained so they must talk to the gateway proxy for outbound access.
- **RBAC**: the gateway is granted the minimum Kubernetes permissions needed to create/delete worker resources.
- **Runtime hardening**: Lobu is designed to run with stronger isolation runtimes when available:
  - **gVisor** (e.g. GCP)
  - **Kata Containers** / microVMs (Firecracker) where available
- **PVC per session**: each session can get its own persistent volume so workspaces survive restarts while remaining isolated.

## Docker Compose

In Docker Compose mode, the gateway uses the local Docker socket to spawn per-session worker containers.

Controls typically used:
- **Internal network for workers**: workers run on an internal Docker network and are not exposed to the host network directly.
- **Gateway as the only egress**: workers are configured to use the gateway's HTTP proxy for outbound requests.
- **Scoped workspaces**: each worker uses its own workspace volume so session state persists but does not mix across sessions.

## Network Egress

Workers do not have direct internet access. Outbound traffic is routed through the gateway's HTTP proxy, which enforces:
- **Allowlist mode**: only configured domains are reachable.
- **Blocklist mode**: allow all except denied domains.

This is a core part of how Lobu makes "agentic" code execution safer in multi-tenant environments.

## Credentials

Lobu centralizes credential flows:
- Provider credentials and client secrets are stored/managed on the gateway side.
- MCP server credentials are resolved per-user via the device-auth flow and injected by the gateway proxy.
- Third-party API auth (GitHub, Google, etc.) is handled by Owletto — workers access these through Owletto MCP tools.
- Workers receive only the minimum scoped tokens required to perform an action, and only via the proxy path.

Result: a compromised worker session should not leak global platform tokens or MCP client secrets.

## Skills And Policy

Skills extend agent capability and should be treated as executable, security-sensitive input.

Recommended stance:
- Use curated skill lists by default.
- Apply tool and network policy (and any internal allow/deny rules) consistently across runtimes.
- Prefer reproducible environments (e.g. Nix) over ad-hoc installs when possible.
